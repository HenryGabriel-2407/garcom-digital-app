1. VISAO GERAL DO PROJETO
================================================================================

SISTEMA: Sistema de Gestao Pizzaria Forno di Resistenza
TIPO: Sistema integrado (App Mobile + API REST)
PUBLICO-ALVO: Pizzaria com atendimento presencial e delivery

OBJETIVO:
  Sistema completo de gestao para pizzaria, abrangendo cadastro de produtos,
  combos e categorias; gerenciamento de mesas e comandas (pedidos); controle
  de cozinha com atualizacao de status em tempo real; cadastro de clientes
  e funcionarios; codigos promocionais; metodos de pagamento; auditoria de
  operacoes; e fluxo de delivery (pedidos online).

USUARIOS E PAPEIS:
  - Garcom:  Atendimento presencial (abrir comandas, adicionar itens,
             finalizar pedidos)
  - Cozinha: Visualizar e atualizar status dos pedidos (em preparo, pronto)
  - Admin:   Gestao completa (CRUD de produtos, funcionarios, mesas,
             clientes, promocoes, visualizar auditoria)
  - Gerente: Visualizar auditoria (acesso similar ao admin mas restrito)
  - Cliente: Realizar pedidos (fluxo futuro via QR code da mesa)

TECNOLOGIAS UTILIZADAS:

  APLICATIVO MOBILE (FRONTEND):
  - React Native com Expo (SDK ~51)
  - TypeScript
  - expo-router (roteamento baseado em arquivos)
  - Axios (cliente HTTP)
  - AsyncStorage (armazenamento local de token JWT)
  - expo-print / expo-sharing (geracao de PDF de recibos)
  - react-native-qrcode-svg (geracao de QR code)
  - @react-native-community/datetimepicker (selecao de datas)
  - react-native-safe-area-context

  API BACKEND:
  - Python 3.12+
  - FastAPI (framework web)
  - SQLAlchemy 2.0 (ORM)
  - Pydantic v2 (schemas e validacao)
  - PyJWT (tokens JWT)
  - pwdlib (hashing de senhas com bcrypt/argon2)
  - Alembic (migracoes de banco)
  - Postgres (banco de dados)
  - aiosmtplib (envio de e-mail)
  - Pydantic Settings (configuracao via .env)
  - pytest + httpx (testes automatizados)

ARQUITETURA RESUMIDA:
  +------------------+       HTTP (REST)       +------------------+
  |  App Mobile      | ----------------------->|  API FastAPI      |
  |  (React Native)  | <-----------------------|  (Python)         |
  +------------------+       JSON + JWT        +------------------+
                                                     |
                                                     v
                                              +------------------+
                                              | Postgres Database |
                                              |  (SQLAlchemy)     |
                                              +------------------+

  O aplicativo mobile consome a API REST via chamadas HTTP com autenticacao
  JWT (Bearer token). O token e obtido no login e armazenado no AsyncStorage,
  sendo automaticamente anexado as requisicoes via interceptor do Axios.
  A URL base da API e descoberta dinamicamente a partir do hostUri do Expo.

5. OBSERVACOES TECNICAS RELEVANTES
================================================================================

--------------------------------------------------------------------------------
5.1. AUTENTICACAO
--------------------------------------------------------------------------------

  - Mecanismo: JWT (JSON Web Token) via OAuth2 Password Bearer flow
  - Dois tipos de usuario: Cliente e Funcionario (unificados no mesmo
    endpoint /auth/token e no mesmo dependency get_current_user)
  - Token payload: { sub: email, tipo: "cliente"|"funcionario", cargo? }
  - Token armazenado no AsyncStorage do app (chave: @access_token)
  - Interceptor Axios: le o token e anexa como Authorization: Bearer
  - SignOut: limpa AsyncStorage (@access_token, @user_data) e remove
    header padrao do Axios
  - Configuracoes: SECRET_KEY, ALGORITHM, ACCESS_TOKEN_EXPIRE_MINUTES via .env

--------------------------------------------------------------------------------
5.2. SEGURANCA
--------------------------------------------------------------------------------

  - Senhas hasheadas com pwdlib (bcrypt/argon2 adaptativo)
  - Controle de acesso por decorator/verificacao no codigo:
      * Publico: sem auth
      * Qualquer autenticado (cliente ou funcionario)
      * Cliente apenas: GET /clientes/me, alterar-senha
      * Funcionario apenas: metodos-pagamento CRUD, mesas CRUD,
        comandas PUT/DELETE/enviar-recibo
      * Admin apenas: produtos CRUD, combos CRUD, categorias CRUD,
        funcionarios CRUD, clientes list/delete/ativar
      * Admin ou Gerente: auditoria logs
      * Proprio usuario: atualizar/visualizar proprio perfil
  - CORS configurado para allow_origins=["*"]

--------------------------------------------------------------------------------
5.3. AUDITORIA
--------------------------------------------------------------------------------

  - Funcao audit.log_audit() registra operacoes em tabela audit_log
  - Captura: usuario_tipo, usuario_id, acao, tabela_afetada, registro_id,
    dados_anteriores, dados_novos, IP, User-Agent
  - Eventos auditados: produto CRUD, combo CRUD (+ associacoes),
    cliente CRUD (+enderecos, +senha), comanda CRUD (+itens, +status)
  - Acesso: apenas admin e gerente
  - Logs nao sao sobrescritos - append-only

--------------------------------------------------------------------------------
5.4. MAQUINA DE ESTADOS DA COMANDA
--------------------------------------------------------------------------------

  Transicoes validas:
    aberta --------> em_preparo (cozinha inicia)
    aberta --------> cancelada (garcom ou cliente)
    em_preparo ----> pronta (cozinha finaliza)
    em_preparo ----> cancelada
    em_preparo ----> aberta (reversao ao editar item)
    pronta --------> entregue (garcom entrega)
    pronta --------> cancelada
    pronta --------> paga (finalizacao)
    pronta --------> aberta (reversao)
    entregue ------> paga (finalizacao)
    cancelada -----> (terminal)
    paga ----------> (terminal)

  Efeitos colaterais ao pagar:
    - Mesa liberada (status -> livre)
    - E-mail de recibo enviado ao cliente
    - Status_pagamento -> pago

  Acoes que revertem em_preparo/pronta para aberta:
    - Adicionar item
    - Editar item (quantidade/observacao)
    - Remover item

--------------------------------------------------------------------------------
5.5. PADROES DE PROJETO NO APP
--------------------------------------------------------------------------------

  - useFocusEffect: todas as telas de lista usam este hook para
    recarregar dados ao ganhar foco (garante dados atualizados)
  - useCallback: funcoes de fetch envolvidas em useCallback para
    evitar re-renderizacoes desnecessarias
  - useRef: usado para debounce de busca (nova-comanda) e cache de
    detalhes de produtos/combos (cozinha)
  - Navegacao por parametros de rota: telas passam dados entre si
    via params do expo-router (ex: cardapio -> nova-comanda, comanda)
  - AsyncStorage: apenas para token JWT e dados do usuario logado
  - Role-based routing: redirect em app/index.tsx + visibilidade de
    abas em (tabs)/_layout.tsx

--------------------------------------------------------------------------------
5.6. BIBLIOTECAS NOTAVEIS
--------------------------------------------------------------------------------

  Axios:
    - Cliente HTTP unico (services/api.ts)
    - Request interceptor para anexar JWT automaticamente
    - Base URL dinamica via Constants.expoConfig.hostUri

  expo-print + expo-sharing:
    - Usado em ComandaScreen e AdminComandasScreen
    - Gera HTML de recibo, converte para PDF via Print.printToFileAsync
    - Compartilha via Sharing.shareAsync

  react-native-qrcode-svg:
    - Usado em QRCodeScreen
    - Renderiza QR code a partir de string URL

  @react-native-community/datetimepicker:
    - Usado em AdminComandasScreen
    - Alterna entre modais nativas do Android e iOS
    - Tratamento especial para iOS (modal vs inline)

--------------------------------------------------------------------------------
5.7. TRATAMENTO DE ERROS
--------------------------------------------------------------------------------

  App Mobile:
    - Erros de API capturados em try/catch
    - Exibidos via Alert.alert() ou variavel de estado error
    - Silencio em alguns casos (ex: admin/auditoria falha sem alerta)
    - Loading states controlados por booleanos

  API:
    - Erros de validacao: HTTP 422 (Pydantic)
    - Nao encontrado: HTTP 404 (excpetion customizada)
    - Nao autorizado: HTTP 401 (JWT invalido/expirado)
    - Proibido: HTTP 403 (cargo sem permissao)
    - Conflito: HTTP 409 (ex: produto vinculado a combo)
    - Erro interno: HTTP 500
    - Respostas padrao: MessageResponse { message, success }

--------------------------------------------------------------------------------
5.8. MODELOS DE DADOS - RESUMO
--------------------------------------------------------------------------------

  15 tabelas no banco SQLite:
    categoria_produto   - Categorias de produtos
    produto             - Produtos do cardapio
    combo               - Combos promocionais
    combo_produto       - Associacao N:N entre combo e produto
    metodo_pagamento    - Formas de pagamento (dinheiro, cartao, pix...)
    cliente             - Clientes cadastrados
    endereco_cliente    - Enderecos dos clientes (1:N)
    funcionario         - Funcionarios (garcom, cozinha, admin, gerente)
    mesa                - Mesas do salao
    comanda             - Pedidos (comandas abertas/finalizadas)
    pedido_item         - Itens de cada comanda (N:1)
    status_comanda_log  - Historico de status das comandas
    cod_promocional     - Codigos de cupom de desconto
    audit_log           - Auditoria de operacoes
    password_reset_token - Tokens de redefinicao de senha

  Principais relacionamentos:
    - produto N:1 categoria_produto
    - produto N:N combo (via combo_produto)
    - endereco_cliente N:1 cliente
    - comanda N:1 cliente, N:1 mesa, N:1 funcionario (garcom)
    - comanda N:1 metodo_pagamento, N:1 cod_promocional
    - pedido_item N:1 comanda, N:1 produto?, N:1 combo?
    - status_comanda_log N:1 comanda
    - audit_log N:1 funcionario
